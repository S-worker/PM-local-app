Guide API Securichat (Prisme.ai)

URL de base : https://api.cagip-securichat-prd.aws.cagip.gca/v2
Documentation générale Prisme.ai : https://docs.prisme.ai/api-reference/sdks

1. Authentification

Toutes les requêtes doivent inclure le token d'accès dans le header Authorization.

export SECURICHAT_TOKEN="ton_token_ici"

curl -X GET "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN"

3. Exemples d'utilisation

3.1 Lister les workspaces

curl -s -X GET "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN"

3.2 Créer un workspace

curl -s -X POST "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -d '{"name": "Test Workspace"}'

Réponse :

{"name":"Test Workspace","id":"gGh0Udq","slug":"cowardly-cougar-16"}

3.3 Modifier un workspace

curl -s -X PATCH "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces/gGh0Udq" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -d '{"description": "Workspace de test"}'

3.4 Créer une page

curl -s -X POST "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces/gGh0Udq/pages" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -d '{"name": "Test Page"}'

Réponse :

{"name":"Test Page","slug":"mighty-termite-21","id":"NGyK2lD","workspaceId":"gGh0Udq"}

3.5 Créer une automation / agent

curl -s -X POST "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces/gGh0Udq/automations" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -d '{"name": "Test Automation", "do": []}'

Réponse :

{"name":"Test Automation","do":[],"slug":"Test Automation-1"}
Attention : le champ do est obligatoire. C'est le DSL (graphe d'actions) de l'automation Prisme.ai.

3.6 Rechercher des utilisateurs par ID

curl -s -X PATCH "https://api.cagip-securichat-prd.aws.cagip.gca/v2/users" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -d '{"query": {"ids": ["690b11d1bc549bb3a28377c6"]}, "data": {}}'

3.7 Rechercher des utilisateurs par email

curl -s -X PATCH "https://api.cagip-securichat-prd.aws.cagip.gca/v2/users" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -d '{"query": {"emails": ["prenom.nom@cagip.fr"]}, "data": {}}'
Blocage connu : le champ data de PATCH /v2/users est obligatoire et non vide, mais son schéma est inconnu. Il refuse les champs testés : firstname, role, roles, permissions, workspaces, tags. Même data: {} retourne data body must not be empty. Contactez l'équipe Securichat pour obtenir le schéma exact.

3.8 Supprimer une page

curl -s -X DELETE "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces/gGh0Udq/pages/mighty-termite-21" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN"
Attention : l'API utilise le slug pour identifier les pages et les automations, pas leur id.

3.9 Supprimer une automation

curl -s -X DELETE "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces/gGh0Udq/automations/Test%20Automation-1" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN"

3.10 Supprimer un workspace

curl -s -X DELETE "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces/gGh0Udq" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN"

7. Authentification et renouvellement du token

7.1 Type de token

Le token utilisé lors de l'exploration était un access token. Il permet :

  • La lecture des workspaces.
  • La création/modification/suppression de workspaces, pages et automations.
  • La recherche d'utilisateurs.

Certaines opérations (comme la mise à jour avancée des utilisateurs via PATCH /users) peuvent nécessiter un autre type de token :

  • Service token : pour les appels machine-to-machine.
  • Admin token : pour la gestion des utilisateurs et des permissions.
  • Session JWT : pour les actions liées à la session utilisateur connecté.

7.2 Stockage sécurisé du token

Ne jamais :

  • Commiter le token dans un dépôt Git.
  • L'écrire en dur dans un script partagé.
  • L'afficher dans des logs ou des captures d'écran.

Privilégier :

  • Les variables d'environnement.
  • Les vaults (HashiCorp Vault, AWS Secrets Manager, etc.).
  • Les fichiers de configuration avec permissions restreintes.

7.3 Renouvellement

Si le token expire, il faut le régénérer depuis l'interface Securichat ou contacter l'équipe responsable. La procédure exacte dépend de la configuration SSO/CAGIP en place.

# Vérifier si le token est encore valide
curl -s -X GET "https://api.cagip-securichat-prd.aws.cagip.gca/v2/workspaces" \
  -H "Authorization: Bearer $SECURICHAT_TOKEN" \
  -o /dev/null -w "%{http_code}\n"

# Si le code retour est 401, le token est expiré ou invalide.

9. Comparaison avec la documentation officielle Prisme.ai

L'instance cagip-securichat-prd est une instance customisée de Prisme.ai. Elle n'expose pas tous les endpoints documentés officiellement.

Endpoint officiel Prisme.aiÉtat sur Securichat CAGIPCommentaire
GET /v2/auth/me❌ Route not foundProfil connecté non exposé.
POST /v2/auth/login❌ Route not foundAuthentification gérée par le SSO CAGIP.
GET /v2/agents❌ Route not foundLes agents sont des automations dans les workspaces.
GET /v2/threads❌ Route not foundEndpoint non exposé.
GET /v2/conversations❌ Route not foundEndpoint non exposé.
GET /v2/workspaces✅ FonctionnelListe des workspaces accessibles.
POST /v2/workspaces/:id/automations✅ FonctionnelCréation d'automations.
PATCH /v2/users⚠️ PartielRecherche OK, schéma data inconnu.
Conclusion : l'API Securichat CAGIP est une version restreinte de l'API Prisme.ai. Pour toute opération avancée, il est recommandé de demander la documentation spécifique de l'instance ou un compte avec des droits élevés.

11. Recommandations pour le codage agentique

Pour commencer :
  • Utilisez le script Python ou le manifeste YAML pour créer des agents de test.
  • Versionnez vos définitions d'agents dans Git (Infrastructure as Code).
  • Créez un workspace par environnement (dev, recette, prod).
Pour aller plus loin :
  • Contactez l'équipe Securichat pour obtenir la documentation du DSL do et les endpoints d'exécution conversationnelle.
  • Demandez un service token ou un token admin si vous devez gérer des utilisateurs.
  • Vérifiez le SDK / package Artifactory adapté à votre langage (Python, Node.js, Java).
Limites actuelles :
  • Pas d'API confirmée pour les conversations / threads.
  • PATCH /v2/users non utilisable sans le schéma exact du champ data.
  • Pas d'endpoint de healthcheck.

12. Prochaines étapes recommandées

  1. Contacter l'équipe Securichat pour obtenir la documentation OpenAPI / Swagger de l'instance cagip-securichat-prd.
  2. Clarifier le schéma du champ data pour PATCH /v2/users.
  3. Vérifier les permissions associées au token (workspaces accessibles, rôle utilisateur).
  4. Choisir le SDK correspondant au langage du projet et à la version de l'API v2.
Points de vigilance :
  • Ne pas exposer le token dans des logs ou des scripts partagés.
  • Les slugs d'automations contenant des espaces doivent être URL-encodés (%20).
  • Supprimer les objets de test après utilisation.
  • La suppression d'un workspace supprime automatiquement ses pages et automations.
